# 摄像头偏移检测与关键区域自适应 — 项目 Handoff

> 这份文档是从一次需求/选型对话沉淀下来的结论，作为本工作空间 (`/home/owen/ORB`) 的项目上下文。
> 代码还没开始写，本文件先定方向、定方案、定边界。

## 项目背景与目标

物业安防摄像头通常**固定**，但可能因环境原因（风吹、震动、被碰）发生**旋转**——注意是**绕 x/y/z 三个轴任意叠加的旋转**（pan/tilt/roll），**不只是画面内的平面旋转**；极少数情况下发生**小幅平移**。
摄像头画面接入 **YOLO** 做目标检测：每个摄像头由人工画一个多边形「**关键区域**」，YOLO 在区域内检出目标即告警。

**痛点**：摄像头一旦旋转，人工画的多边形在画面里就「漂」到了别的真实位置 → **误报**。

**要解决的三件事**：
1. **偏移检测**：摄像头偏离标准位时能检测到并告警。
2. **角度估计**：估算出旋转/偏移的角度。
3. **多边形自适应**：自动校正多边形，使其**始终贴合人为设定的真实世界区域**。

## 核心洞察（整个方案的地基）

这**不是**一个检测/AI 问题，本质是**图像配准 (image registration)**。

关键数学事实：当相机**绕光心旋转**时，旋转前后两帧之间的映射**严格是一个单应矩阵 H**，且**与场景深度无关**（这正是全景拼接的原理）：

```
H = K · R · K⁻¹      （K=相机内参, R=旋转矩阵）
```

**这里的 R 是完整的 3D 旋转矩阵 (SO(3))，绕 x/y/z 任意叠加都包含在内**，不是只有绕光轴的平面旋转。
绕 x/y 轴的旋转（pan/tilt）会在画面里产生**透视/梯形畸变 (keystone)**，而完整 8 自由度的 H 能正确表达这种畸变——
**这正是必须用单应、而不能用仿射/相似变换抄近路的根本原因。**

含义：**只要估出 H，就能把人工画的多边形点 p 搬到当前画面的正确位置 p' = H·p。**
→ 三个需求（检测偏移 / 算角度 / 自适应多边形）**用「估计 H」这一个机制统一解决**，不需要三套逻辑。

> ⚠️ 例外：**平移**会产生视差，单个 H 不再严格成立。但平移是极少数且幅度小，H 仍是好近似；
> 关键区域多贴着地面/某平面，可对该平面拟合「地面单应」，对那个平面依然精确。
> **大幅平移/视差过大 → 走告警让人复位，不要硬套 H。**

## 整体架构

**① 标定阶段（装机、画多边形时做一次）**，保存：
- 标准位置参考帧 `ref_frame`
- 参考帧特征点 + 描述子（预先算好缓存）
- 多边形坐标（参考帧坐标系下）

**② 运行阶段（不必每帧，每隔 N 秒或定时触发；用缩小图，开销很小）**：
```
当前帧 → 提特征 → 与参考帧匹配 → RANSAC 估 H
       ├─ 算偏移量 → 超阈值告警            (需求1)
       ├─ 分解 H 得旋转角                 (需求2)
       └─ 多边形 = perspectiveTransform(多边形_ref, H)   (需求3)
```
YOLO 照常每帧跑；偏移检测低频跑，互不影响。

## 三个需求的落地方式

**需求 1 — 偏移检测与告警**
用 H 投影参考帧四角/中心，看移动了多少像素；或看 RANSAC 内点匹配的中位位移。超阈值 → 「偏离标准位」。
*兜底*：匹配内点过少 / H 求解失败 → 画面变化过大（大幅平移、遮挡、夜间失配），单独告警人工介入，**不要硬套 H**。

**需求 2 — 角度估计（三个轴，不只一个角）**
旋转是三轴叠加，要报 **pan / tilt / roll 三个角**。从 `R = K⁻¹·H·K` 分解（结果再正交化）得到旋转矩阵 → 欧拉角。
**本项目无法用棋盘格标定 K**（见下方约束）→ 采用：
- **自标定**：纯旋转相机可从单帧/多帧的 H 反推焦距 f（假设主点在中心、方形像素），Hartley 全景自标定思路；
- 或用画幅 + 假定 FOV 粗估 f。
- **roll**（绕光轴/画面内旋转）即使没有 K 也能直接近似：`atan2(H[1,0], H[0,0])`。
- **pan/tilt**（绕 x/y 轴）**必须知道 f 才能分离出真实角度**；f 估不准时退化为「像素漂移量」报警，仍可用。
> 注：`cv2.decomposeHomographyMat` 是给「平面 + 平移」用的、会给多组解；本项目是**纯旋转**，直接 `R = K⁻¹HK` 再正交化更干净。

**需求 3 — 多边形自适应（全自动）**
`cv2.perspectiveTransform(多边形_ref, H)` 一行得到当前画面的多边形。
**全自动持续校正，必须加保护**（否则某帧 H 估歪会让多边形乱跳）：
- 仅在 H **置信度高**（内点数/重投影误差达标）时才更新；
- 对多边形做**时序平滑**（滑动平均 / 卡尔曼）；
- H 估计失败时**保持上一次的好结果**；
- **大漂移**（疑似平移/被撞）→ 校正之外**额外告警**，让人工确认是否需要重新基准化。

## 已确认的关键约束（来自需求方）

| 点 | 结论 | 对方案的影响 |
|----|------|-------------|
| **昼夜运行** | 需要 7×24，昼夜画面差异大 | **多参考帧（白天/黄昏/夜晚）择优匹配**成为**必需**，不是可选；并用对光照鲁棒的特征/边缘配准 |
| **相机标定** | **不能**用棋盘格标 K，只能从单帧/连续多帧估算 | 需求 2 走**自标定/FOV 估焦**；估不准时退化为像素漂移报警 |
| **场景纹理** | 丰富和低纹理**都可能** | 低纹理时 ORB 会失配 → 需要**降级策略**（学习型匹配 / 边缘配准）+ 失配告警兜底 |
| **多边形校正** | **全自动** | 持续校正 + 置信度门控 + 时序平滑 + 失败回退 + 大漂移告警（见需求3） |

## 昼夜与低纹理的应对（本项目的两大主要风险）

- **昼夜**：单张参考帧扛不住。做法：维护**多张参考帧库**（不同时段/光照），当前帧匹配时择优；特征用 **AKAZE** 或对**边缘/梯度图**做 ECC 配准以抗光照；必要时上学习型匹配。
- **低纹理**：大白墙/纯地面会让 ORB 没特征。降级路线：学习型特征匹配（**SuperPoint+SuperGlue / LoFTR**）或基于强度的 **ECC** 配准；仍失败则**保持上次结果 + 告警**。
- **动态前景**（人/车走动）：**不是问题**——运动物体被 RANSAC 当外点剔除，只要静态背景特征够。

## 推荐技术栈与 v1 范围

- **语言/库**：Python + OpenCV（`cv2.ORB_create`、`BFMatcher`、`findHomography(RANSAC)`、`perspectiveTransform`、`findTransformECC`）。
- **建模硬约束**：三轴旋转有透视畸变，**必须用完整 8-DoF 单应**；禁止用 `estimateAffinePartial2D` 或 ECC 的 `EUCLIDEAN/AFFINE`（只能描述画面内平面旋转，遇到 pan/tilt 会错）。ECC 用 `MOTION_HOMOGRAPHY`。
- **v1（先别过度设计）**：`ORB + BFMatcher + 比率测试 + findHomography(RANSAC)`，单张参考帧缓存。
  跑通主链路：**估 H → 投影多边形 → 偏移超阈值告警**。
  - 验证标准：手动把相机转一个角度，**多边形稳稳跟住真实区域，且报出的角度合理**。
- **v2 升级触发点**：遇到夜间/低纹理失配 → 加多参考帧、AKAZE/ECC、或 SuperPoint+SuperGlue/LoFTR；加自标定出真实角度；加时序平滑与告警分级。
