# 技术设计文档 — 摄像头偏移检测与关键区域自适应

本文描述项目所用的技术、实现方式与具体计算原理，面向维护者与二次开发者。
使用层面的文档见 [`README.md`](README.md)，需求背景见 [`CLAUDE.md`](CLAUDE.md)。

---

## 1. 设计目标与本质判断

要解决三件事：**①偏移检测告警 ②旋转角度估计 ③多边形自适应校正**。

**关键判断：这不是检测/AI 问题，而是图像配准 (image registration) 问题。**
固定相机发生的是**绕光心的三轴旋转**（pan/tilt/roll），旋转前后两帧的映射严格是一个**单应矩阵 H**。一旦估出 H，三个需求用同一个 H 统一解决，不需要三套逻辑：

```
偏移量  = f(H)          投影参考帧角点看位移
角度    = decompose(H)  分解出 pan/tilt/roll
新多边形 = H · 旧多边形   一次透视变换
```

---

## 2. 数学基础

### 2.1 为什么是单应，且与深度无关

设相机内参矩阵 `K`，世界点在两次拍摄间相机发生旋转 `R`（绕光心）。归一化相机坐标下旋转关系为 `x₂ = R·x₁`，乘内参回到像素坐标：

```
p₂ = K·x₂ = K·R·x₁ = K·R·K⁻¹·p₁
        ⇒  H = K · R · K⁻¹        （3×3，8 自由度）
```

推导里**没有出现场景深度 Z**——这正是全景拼接的原理：纯旋转下，单应对**整幅画面所有像素**都成立，与物体远近无关。

### 2.2 为什么必须用完整 8-DoF 单应，不能用仿射/相似

这里的 `R ∈ SO(3)` 是**完整三维旋转**，pan/tilt（绕 x/y 轴）会在画面里产生**透视/梯形畸变 (keystone)**。只有 8 自由度的单应能表达这种畸变；仿射 (6-DoF) / 相似 (4-DoF) 只能描述画面内的平面旋转，遇到 pan/tilt 必错。

> 因此代码里硬约束：`findHomography` 用全 8-DoF，ECC 用 `MOTION_HOMOGRAPHY`，**禁用** `estimateAffinePartial2D` 与 ECC 的 `EUCLIDEAN/AFFINE`。

### 2.3 平移这个例外

纯旋转假设在**平移**时被打破（产生视差，单 H 不再严格）。但物业相机平移极少且幅度小，H 仍是好近似；幅度大时走告警让人复位（见 §5 的大漂移告警），不硬套 H。

## 3. 系统架构

```
① 标定阶段（装机一次, calibrate.py / calibration.py）
   标准位参考帧 ──提 ORB 特征──┐
   人工多边形 ────────────────┼──► cache/camXX.npz
                              （ref_frame + keypoints + descriptors + polygon）

② 运行阶段（定时触发, tracker.py 总集成）
   当前帧 ─► 估计层 ─► H ─► 保护层 ─► 输出
            │                │
            ├ 多参考帧 ORB 择优        ├ 置信度门控
            └ ECC 降级（低纹理/光照）   ├ 时序平滑
                                       ├ 失败回退
   需求1 偏移告警 ◄──┐                  └ 大漂移告警
   需求2 角度    ◄──┼── 都从同一个 H 派生
   需求3 多边形  ◄──┘
```

模块职责：

| 模块 | 职责 |
|------|------|
| `features.py` | ORB 检测器（单点定义，标定/估 H 同参数）+ 关键点序列化 |
| `calibration.py` | 标定数据生成 / `.npz` 存取（低纹理容错） |
| `homography.py` | 估 H 主链路 + 多边形投影 + 内点重投影误差 |
| `angles.py` | 焦距自标定 + 旋转分解 |
| `drift.py` | 偏移量计算与告警 |
| `multiref.py` | 多参考帧库 + 择优 |
| `ecc.py` | 金字塔 ECC 降级 |
| `tracker.py` | 估计降级 + 四道保护，对外唯一入口 `PolygonTracker.update()` |

## 4. 关键算法与计算

### 4.1 特征提取与匹配（ORB 主路径）

`homography.estimate_homography()`：

1. **检测**：`ORB_create(nfeatures=2000)` 在灰度图上 `detectAndCompute`，得到关键点与 32 字节二进制描述子。
2. **匹配**：`BFMatcher(NORM_HAMMING)` 对每个参考特征找当前帧的最近两个 `knnMatch(k=2)`。
3. **比率测试 (Lowe)**：仅保留 `最近距离 < 0.75 × 次近距离` 的匹配，剔除歧义匹配。
4. **RANSAC 估 H**：`findHomography(src, dst, RANSAC, 3.0)`。RANSAC 自动把**动态前景**（行人、车辆）和误匹配当作外点剔除，只用静态背景内点拟合，这就是动态前景"不是问题"的原因。
5. **内点重投影误差**：把内点 `src` 经 H 投到当前帧，与真实 `dst` 求平均 L2 距离：

```
reproj_err = mean‖ perspectiveTransform(src_inlier, H) − dst_inlier ‖
```

输出 `HomographyResult(H, num_good, num_inliers, ok, reproj_err)`，其中 `inlier_ratio = num_inliers / num_good`。这三个量（内点数、比率、重投影误差）就是后续置信度门控的依据。

匹配/内点不足（<4）或 `findHomography` 返回 None → `ok=False`，交上层降级或告警，**不硬套 H**。

### 4.2 多边形投影（需求 3 的核心一行）

```python
poly_cur = cv2.perspectiveTransform(poly_ref.reshape(-1,1,2), H)
```

即对每个顶点做 `p' = H·p`（齐次坐标，再除以第三分量）。

### 4.3 偏移量计算（需求 1）

`drift.corner_displacements()`：取参考帧四角 `(0,0),(w,0),(w,h),(0,h)`，经 H 投影后与原位置求欧氏距离：

```
disp_i = ‖ H·corner_i − corner_i ‖ ,  i=0..3
偏移量 = max(disp)   （也报 mean）
```

`drift.evaluate()` 的判定：H 失败/内点过少 → 告警人工介入；否则 `max_disp > 阈值` → 偏离标准位告警。用四角而非中心，是因为 pan/tilt 的梯形畸变在角部最明显。

### 4.4 焦距自标定（需求 2 的前置）

本项目**无法用棋盘格标定 K**，需从单帧 H 反推焦距 `f`（假设主点在画面中心、方形像素，即 `K = diag(f, f, 1)`）。

**第一步：把 H 共轭到主点在原点的坐标系**（自标定公式的前提）。设平移 `T` 把中心移到原点：

```
Hc = T⁻¹ · H · T ,   T = [[1,0,cx],[0,1,cy],[0,0,1]]
```

**第二步：用旋转矩阵的正交约束解 f²。** 由 `Hc = K R K⁻¹` 知 `R = K⁻¹ Hc K` 必须正交。把这一约束（行/列单位正交）展开，可得到两组关于 `f²` 的闭式解（OpenCV `focalsFromHomography` 思路，记 `Hc` 行优先展开为 `h0..h8`）：

```
解①（由第三列约束）:
  d1 = h6·h7
  d2 = (h7−h6)(h7+h6)
  v1 = −(h0·h1 + h3·h4) / d1
  v2 = (h0²+h3² − h1²−h4²) / d2
解②（由第三行约束）:
  d1 = h0·h3 + h1·h4
  d2 = h0²+h1² − h3²−h4²
  v1 = −h2·h5 / d1
  v2 = (h5²−h2²) / d2

每组在两个候选 (v1,v2) 中取正者（分母绝对值更大的更可信），f = √(f²)。
最终 f = 两组有效解的中位数。
```

**关键退化：纯画面内旋转 (只有 roll) 时 `h6=h7≈0` 且 `h0h3+h1h4≈0`，分母为 0，f 无法求出。** 这是数学本质而非 bug——画面内旋转不含透视信息，自然解不出焦距。此时退化为只报 `roll`（见 4.5）。真实相机的 pan/tilt 带来透视畸变，f 才估得出。

### 4.5 旋转分解 pan / tilt / roll（需求 2）

**角度约定**：`R = Rz(roll) · Ry(pan) · Rx(tilt)`，pan 绕 y（左右摇）、tilt 绕 x（俯仰）、roll 绕 z（画面内旋转）。

**roll 的免标定近似**（不依赖 f，f 估不准时的兜底）：

```
roll ≈ atan2(H[1,0], H[0,0])
```

**完整分解**（f 可用时）：

```
1. K = diag(f, f, 1)
2. R = K⁻¹ · Hc · K              （Hc 为居中后的 H）
3. 正交化：SVD(R)=U·S·Vᵀ ⇒ R ← U·Vᵀ；若 det(R)<0 翻转 U 末列
   （H 含噪声/尺度，R 未必严格正交，SVD 取最近的合法旋转矩阵）
4. 按约定逆解欧拉角：
     pan  = atan2(−R[2,0], hypot(R[0,0], R[1,0]))
     roll = atan2( R[1,0], R[0,0])
     tilt = atan2( R[2,1], R[2,2])
```

> 用 `R=K⁻¹HcK` 直接分解，而非 `cv2.decomposeHomographyMat`——后者是给"平面+平移"用的、会返回多组解；本项目是纯旋转，直接解更干净唯一。

### 4.6 ECC 降级（低纹理 / 强光照）

ORB 在大白墙、纯地面或光照剧变下匹配不足时，`ecc.estimate_ecc()` 接管：基于整图**灰度强度**做对齐，不需要特征点。

**为什么喂灰度而不是梯度图**：ECC（增强相关系数）的判据本身对亮度/对比度的仿射变化天然不变——这就是它"抗光照"的来源。实测喂稀疏的梯度图反而会让其高斯牛顿迭代发散，故直接喂灰度 float32（ECC 内部 `gaussFiltSize` 再做平滑）。

**金字塔多尺度（粗→细）**，扩大收敛域（单尺度只能纠正几像素错位）：

```
for 层 lvl = 顶(最粗) → 底(原分辨率):
    s = 0.5^lvl ;  S = diag(s, s, 1)
    W = S · H · S⁻¹                       # 把当前 H 缩放到该层尺度
    cc, W = findTransformECC(refₗ, curₗ, W, MOTION_HOMOGRAPHY)
    H = S⁻¹ · W · S                       # 还原回原分辨率
```

`S H S⁻¹` 是相似变换下单应的尺度变换：坐标缩放 `s` 时，平移分量随之缩放，透视分量反向缩放。

**逐层容错**：过平滑的粗层可能让 `findTransformECC` 抛异常，此时跳过该层、保留当前 H 继续更细层，而不是整体失败。返回 `(H, cc)`，`cc∈[-1,1]` 作为该路径置信度（默认 `≥0.6` 才采用）。

### 4.7 多参考帧择优（昼夜）

`multiref.estimate_best()`：对库中每张参考帧各跑一次 4.1 的 ORB 估计，返回**内点数最多**的结果及其索引。夜间帧会自然匹配到夜间 ref（内点多），白天 ref 内点寥寥而被淘汰。单张参考帧场景在 `tracker` 内部被包成"1 张的库"，上层逻辑统一。

### 4.8 时序平滑（EMA）

每帧投影出的 `raw` 多边形可能因 H 噪声轻微抖动。对顶点做指数滑动平均：

```
smoothed ← α · raw + (1−α) · smoothed     （默认 α=0.3）
```

α 越小越平滑、越滞后。顶点一一对应（同一多边形、同序），逐点平滑。

## 5. 保护逻辑（全自动校正的安全网）

`PolygonTracker.update()` 的状态机——全自动持续校正必须有保护，否则某帧 H 估歪会让多边形乱跳：

```
                 ┌─────────────────────────────────────────┐
当前帧 ─► 估计层 ─►│ 多参考帧 ORB 择优 ─► 置信？               │
                 │   否 ─► ECC 降级 ─► cc≥阈值？             │
                 └──────────────┬──────────────────────────┘
              置信(orb/ecc)     │   都不达标
                    │           ▼
                    ▼     ┌──────────────────────────┐
        ┌──────────────┐  │ 保持上次好结果(不更新)     │
        │ 投影 + EMA 平滑│  │ + 告警"需人工介入"        │
        │ 更新 last_good│  └──────────────────────────┘
        │ 算角度        │
        │ 偏移>大阈值?  │──► 额外告警"疑似平移/被撞"
        └──────────────┘
```

四道保护对应代码：

1. **置信度门控** `_orb_confident()`：
   ```
   ok ∧ num_inliers ≥ 40 ∧ inlier_ratio ≥ 0.5 ∧ reproj_err ≤ 3px
   ```
   ECC 路径用 `cc ≥ 0.6` 作为独立门控。
2. **时序平滑**：见 4.8，仅在置信时更新平滑值。
3. **失败回退**：不置信时直接返回上一次的 `smoothed`，多边形原地不动。
4. **大漂移告警**：即便置信，若 `max_disp > big_drift_thresh(60px)`，在正常校正之外**额外**告警，提示疑似平移/被撞、人工确认是否重新基准化。

## 8. 性能考量

- 偏移检测**不必每帧跑**，定时/每隔 N 秒触发即可，建议用**缩小图**——单应是尺度可换算的，缩小后估的 H 用 §4.6 同样的 `S H S⁻¹` 变换还原。
- ORB+RANSAC 在缩小图上是毫秒级；ECC 金字塔较重，只在 ORB 降级时才触发。
- 多参考帧把单帧开销乘以库大小（N 张 ref → N 次 ORB 匹配），库不宜过大（建议 3–4 张覆盖白天/黄昏/夜晚）。

---

## 9. 设计取舍与已知局限

- **角度符号需现场标定一次**：图像 y 轴朝下，分解出的角度符号方向需拿真实相机转一个已知方向核对，确保告警的"左偏/右偏"语义正确。
- **纯画面内旋转无法分离 pan/tilt**（§4.4），退化为像素漂移+roll 报警，可用但信息少。
- **大幅平移/视差**超出单应模型，靠大漂移告警兜底，不强行校正。
- **强空间不均光照下 ECC 的 cc 偏低**（对齐其实准但门控偏保守），昼夜主力应是多参考帧而非 ECC。

## 10. 扩展点（v2）

- **学习型匹配**：夜间/低纹理仍失配时，用 SuperPoint+SuperGlue / LoFTR 替换 §4.1 的 ORB 估计层；保护层（§5）完全不动。
- **多边形卡尔曼平滑**：把 §4.8 的 EMA 升级为卡尔曼，更好地处理快速但平滑的偏移。
- **地面单应**：若关键区域贴地，可对地平面拟合"地面单应"，在小幅平移下对该平面仍精确。
